# 代码重构Skill
## Overview
本技能专注于为各类项目（Web应用、后端服务、移动端代码、工具类项目等）提供全流程代码重构支持，
核心价值是在不改变代码原有业务逻辑的前提下，优化代码结构、提升代码可读性、降低维护成本、消除技术债务，
适配不同技术栈（Java、Python、Go、JavaScript等）、不同规模（小型工具、中型项目、大型系统）的代码重构需求，
兼顾重构效率与代码稳定性，避免重构过程中引入新的bug。

## When to Use
✅ 适用场景：
1.  代码冗余严重，存在大量重复代码、无效代码，维护成本高；
2.  代码可读性差，命名不规范、注释缺失、逻辑混乱，新人上手困难；
3.  代码性能瓶颈，存在低效循环、冗余计算、资源泄漏等问题，影响系统运行速度；
4.  技术债务堆积，旧代码不符合当前技术规范，需适配新框架、新依赖；
5.  代码耦合度高，模块间依赖混乱，难以扩展和迭代；
6.  代码不符合行业规范或团队编码标准，需统一整改。

❌ 不适用场景：
1.  改变原有业务逻辑的代码修改（重构核心是“不改变功能”，仅优化结构）；
2.  全新代码编写（仅针对已有代码的优化，不负责从零开发）；
3.  代码调试、bug修复（仅优化代码结构，不解决业务逻辑层面的bug）；
4.  非代码层面的优化（如服务器配置、数据库优化，仅聚焦代码本身）；
5.  无明确重构目标或未提供相关代码片段/代码现状的需求。

## Workflow（必须按序执行）
1.  需求拆解：询问用户获取核心信息（代码技术栈、重构目标、代码规模、业务场景、团队编码规范、重构周期）；
2.  现状分析：查看用户提供的代码片段/项目结构，识别代码问题（冗余、耦合、性能、规范性等），梳理技术债务；
3.  方案设计：制定分阶段重构方案，明确重构优先级、核心优化点、风险点及应对措施，确定重构工具（如代码检测工具、格式化工具）；
4.  执行指导：提供具体重构步骤，包括代码修改方法、命名规范调整、注释补充、模块拆分/合并、冗余代码删除等；
5.  验证测试：指导用户进行重构后测试（单元测试、集成测试），确保重构后代码功能与原有一致，性能/可读性达标；
6.  文档输出：整理重构报告，包含重构前后对比、优化亮点、后续维护建议，补充代码注释、文档说明。

## Examples
### 示例1：输入（代码冗余重构需求）
"现有Python后端代码，存在大量重复的数据库操作函数，命名不规范，注释缺失，技术栈是Python+Django，需要重构优化，提升可读性和可维护性。"

### 示例1：输出（核心重构方案）
#### 一、重构目标
1.  消除重复代码，封装公共数据库操作函数；
2.  统一命名规范，符合Python PEP8标准；
3.  补充必要注释，提升代码可读性；
4.  梳理函数依赖，降低耦合度。

#### 二、现状问题分析
1.  重复代码：多个模块存在相同的数据库查询、插入、更新函数，未进行封装；
2.  命名不规范：函数名大小写混乱（如getuser、GetUser），变量名模糊（如a、b、data1）；
3.  注释缺失：核心逻辑、参数说明、返回值说明未标注，新人难以理解；
4.  依赖混乱：部分函数依赖全局变量，可移植性差。

#### 三、具体重构步骤
1.  封装公共函数：
    - 在utils文件夹下创建db_utils.py，封装公共数据库操作（如get_one、get_list、insert_data、update_data）；
    - 所有模块调用公共函数，删除重复代码，减少冗余。
2.  规范命名：
    - 函数名采用小写+下划线（snake_case），如get_user、insert_order；
    - 变量名清晰易懂，避免模糊命名，如user_id而非uid、order_list而非data；
    - 类名采用大驼峰（CamelCase），如UserService、OrderDao。
3.  补充注释：
    - 函数头部添加文档字符串（docstring），说明功能、参数、返回值、异常情况；
    - 复杂逻辑处添加行注释，解释核心思路；
    - 标注代码修改记录、作者信息。
4.  解除耦合：
    - 消除全局变量依赖，通过参数传递必要数据；
    - 拆分过大的函数（单个函数代码量不超过100行），按功能拆分多个小函数。

#### 四、验证与工具
1.  工具使用：用pylint检测代码规范，用black格式化代码，确保符合PEP8标准；
2.  测试验证：编写单元测试，覆盖重构后的公共函数和核心模块，确保功能与重构前一致；
3.  性能验证：对比重构前后代码执行效率，确保无性能退化。

### 示例2：输入（代码性能重构需求）
"现有Java接口代码，存在低效循环和冗余计算，接口响应慢，技术栈是Java+Spring Boot，需要优化代码性能，提升接口响应速度。"

### 示例2：输出（核心重构方案）
#### 一、现有问题分析
1.  低效循环：嵌套循环过多（三层及以上），循环内频繁查询数据库、创建对象，导致执行效率低；
2.  冗余计算：多次计算相同的值（如重复获取配置、重复转换数据格式），未进行缓存；
3.  资源浪费：数据库连接未复用，频繁创建和关闭连接；
4.  数据处理方式不当：大量数据在内存中直接处理，未分页、未分批，导致内存占用过高。

#### 二、性能重构方案（分阶段实施）
1.  第一阶段（紧急优化）：
    - 优化循环逻辑：减少嵌套循环，将循环内的数据库查询、对象创建移至循环外；
    - 增加缓存：对频繁查询的配置、静态数据，使用Redis或本地缓存（如Caffeine）缓存，减少重复计算；
    - 复用资源：使用数据库连接池，统一管理连接，避免频繁创建/关闭。

2.  第二阶段（深度优化）：
    - 数据处理优化：大量数据采用分页查询、分批处理，避免一次性加载至内存；
    - 算法优化：替换低效算法（如线性查找改为哈希查找），减少时间复杂度；
    - 异步处理：将非核心逻辑（如日志记录、数据统计）改为异步执行，提升主流程响应速度。

3.  第三阶段（验证与监控）：
    - 性能测试：使用JMeter测试接口重构前后的响应时间、QPS，确保性能提升达标；
    - 监控预警：引入监控工具（如Prometheus），实时监控接口性能，及时发现新的性能瓶颈；
    - 代码复盘：梳理性能优化亮点，形成团队性能优化规范。

#### 三、落地建议
优先实施第一阶段紧急优化，快速提升接口响应速度；第二阶段优化需结合业务场景，避免过度优化；重构过程中做好版本控制，每次重构后及时测试，防止引入新问题。

## 关键规范与注意事项
1.  重构核心原则：**不改变原有业务逻辑**，所有优化仅针对代码结构、性能、可读性，确保重构后功能与原代码一致；
2.  分阶段实施：复杂项目避免一次性大规模重构，分模块、分优先级逐步推进，降低风险；
3.  重视测试：重构前备份代码，重构后必须进行单元测试、集成测试，必要时进行回归测试，避免引入bug；
4.  贴合技术栈：不同技术栈的重构规范不同（如Python遵循PEP8、Java遵循阿里巴巴开发手册），需适配对应规范；
5.  兼顾团队协作：重构方案需符合团队编码标准，便于团队成员理解和维护，避免个人风格主导；
6.  避免过度重构：无需追求“完美代码”，重构以解决实际问题（冗余、性能、维护困难）为目标，兼顾效率与成本。
