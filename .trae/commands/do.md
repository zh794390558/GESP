你现在处于 Do 模式。严格按照已确认的 spec 文档执行编码实现和验证。

---

## 第零步：加载项目上下文

1. 扫描 `.spec/` 目录，列出所有子目录
2. 如果只有一个项目，自动选中；如果有多个，询问用户要执行哪个
3. 读取以下三个文件，作为本次执行的完整上下文：
   - `.spec/<slug>/spec.md` — 变更范围与需求契约
   - `.spec/<slug>/tasks.md` — 执行计划与依赖关系
   - `.spec/<slug>/checklist.md` — 验证清单

读取后，向用户展示任务概况：

- 共几个 Phase、几个 Task、几个 SubTask
- 预计影响哪些文件
- 等待用户确认开始执行

---

## 第一阶段：按 Task Dependencies 约束执行编码

解析 tasks.md 中的 `# Task Dependencies` 章节，构建依赖图，按依赖约束决定执行顺序。

### 执行顺序规则

1. **解析依赖**：从 `# Task Dependencies` 中提取每个 Task 的前置依赖
2. **拓扑排序**：按依赖关系拓扑排序，无依赖的 Task 优先执行，有依赖的 Task 必须等所有前置完成后才能执行
3. **默认回退**：如果 tasks.md 中未声明 Task Dependencies，则按 Phase → Task → SubTask 的书写顺序执行
4. **Phase 内并行**：同一 Phase 内声明可并行的 Task，按书写顺序依次执行（不跳 Phase）

### 依赖约束

- Task 依赖的所有前置 Task 必须全部 `- [x]` 后，本 Task 才可开始
- 如果前置 Task 未完成就遇到本 Task，暂停并告知用户依赖未满足
- 遇到循环依赖时暂停，告知用户，由用户决定如何打破循环

### 每个 SubTask 的执行流程

1. **定位代码**：搜索并读取需要修改的文件和函数，理解当前实现
2. **分析影响**：
   - 查找该函数/变量的所有调用方（向上依赖）
   - 查找该函数/变量依赖的下游（向下依赖）
   - 判断本次修改是否会破坏上下游的接口契约或行为预期
   - 如果会影响上下游，必须一并修改，确保调用链完整一致
3. **制定方案**：说明要改什么、怎么改、为什么这样改、涉及哪些连带修改
4. **编码实施**：执行修改，展示关键变更摘要（不需要贴完整文件，只展示改动部分）
5. **更新状态**：将 `.spec/<slug>/tasks.md` 中对应 SubTask 的 `- [ ]` 更新为 `- [x]`

### 每个 Task 完成后

- 如果该 Task 下所有 SubTask 都已完成，同步将 Task 的 `- [ ]` 更新为 `- [x]`
- 简短汇报进度（已完成 N/M 个 Task，下一批可执行的 Task 是哪些）

### 每个 Phase 完成后

- 汇报本 Phase 改动了哪些文件、是否有连带修改
- 汇报是否有意外发现
- 汇报当前依赖图状态：哪些 Task 已解锁可执行、哪些仍在等待前置
- 将 tasks.md 的状态更新落盘
- 根据依赖图进入下一批可执行的 Task，不暂停

### 影响分析规则

| 场景 | 处理方式 |
|---|---|
| 修改函数签名（参数/返回值） | 搜索所有调用方，逐一确认并修改 |
| 修改变量名或类型 | 搜索所有引用处，逐一确认并修改 |
| 修改控制流（if/else 分支逻辑） | 检查下游对分支结果的依赖是否受影响 |
| 删除代码 | 确认无其他代码依赖被删部分 |
| 新增代码 | 确认与现有代码的交互无冲突 |
| 上下游影响超出本 Task 范围 | 暂停，告知用户，由用户决定是否扩展 Task 范围 |

### 发现新问题

如果在执行过程中发现 spec 未覆盖的 Bug 或需要额外修改：

- **不影响当前流程的小问题**：记录下来，Phase 汇报时一并告知
- **影响当前 Task 正确性的问题**：立即暂停并告知用户，由用户决定是否纳入当前 spec
- **独立的新问题**：记录即可，建议用户单独开 spec 处理

## 第二阶段：Checklist 逐项验证

所有 Phase 执行完成后，进入验证阶段。逐项检查 `.spec/<slug>/checklist.md` 中的每一项，确认代码实现是否正确。

### 验证方法

- **代码级检查**：读取相关代码，确认变更已正确应用
- **逻辑级检查**：走查执行路径，确认 spec 中 WHEN/THEN 契约满足
- **依赖级检查**：确认修改涉及的上下游调用链无断裂、无行为回退
- **语法级检查**：运行 lint、类型检查等（适用时）
- **流程级检查**：确认端到端流程无遗漏

### 逐项输出格式

对每个 checklist 章节和条目，输出：

```
## Bug #N: <标题>
- [x] <条目1> — ✅ <一句话验证结论>
- [x] <条目2> — ❌ <问题描述>
  → 建议：<修复方案>
```

### 判定规则

- **✅ 通过**：代码已正确实现，逻辑无遗漏，上下游完好
- **❌ 不通过**：实现缺失、逻辑错误、与 spec 不一致、或上下游调用链断裂，给出具体建议
- **⚠️ 注意**：实现正确但存在潜在风险（如性能、边界情况），给出提示但不阻塞

### 验证结果处理

1. **全部通过**：将 checklist.md 中所有 `- [ ]` 更新为 `- [x]`，进入第三阶段
2. **存在不通过项**：
   - 列出所有 ❌ 项及建议
   - 询问用户是否需要立即修复
   - 如用户确认修复，回到第一阶段对修复任务编码，修复后重新验证对应条目
   - 如用户选择跳过，在 checklist.md 中标注 `❌` 并附原因注释

## 第三阶段：输出最终摘要

验证完成后，输出实现摘要，包含：

- **变更文件清单**：列出所有修改/新增的文件及简要说明
- **连带修改说明**：超出 spec 原定范围的上下游修改（如有）
- **Task 完成统计**：`N/M SubTask 已完成`
- **Checklist 结果**：`X/Y 项通过，Z 项未通过（如有）`
- **文档路径**：
  - `.spec/<slug>/spec.md`
  - `.spec/<slug>/tasks.md`
  - `.spec/<slug>/checklist.md`

## 异常处理

| 情况 | 处理方式 |
|---|---|
| spec.md 中描述的文件/函数不存在 | 暂停，告知用户，等待确认是路径错误还是需要新建 |
| 执行中发现 spec 设计有缺陷 | 暂停，说明问题，建议回到 Spec 模式修订后再继续 |
| Task 间依赖冲突 | 暂停，说明冲突，由用户决定调整顺序或修改 Task |
| 用户中途要求修改 spec | 暂停当前 Task，回到 Spec 模式修订后重新确认，再继续执行 |
| Do 模式中发现新 Bug | 暂停，告知用户，由用户决定纳入当前 spec 还是单独开 spec |
| 上下游影响超出预期 | 暂停，告知用户影响范围，由用户决定是否扩展 Task |
