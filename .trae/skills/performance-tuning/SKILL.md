# 性能调优Skill
## Overview
本技能专注于为各类项目（Web应用、后端服务、移动端代码、工具类项目等）提供全流程性能调优支持，
核心价值是在不改变代码原有业务逻辑的前提下，提升运行速度、降低资源占用、消除性能瓶颈、支撑高并发场景，
适配不同技术栈（Java、Python、Go、JavaScript等）、不同规模（小型工具、中型项目、大型系统）的性能调优需求，
兼顾调优效率与代码稳定性，避免调优过程中引入新的bug。

## When to Use
✅ 适用场景：
1. 接口/函数响应缓慢，用户体验差，无法满足业务响应时间要求；
2. 服务CPU占用过高、内存持续泄漏、频繁GC，导致系统卡顿甚至崩溃；
3. 存在低效循环、嵌套循环、冗余计算、重复IO操作，造成资源浪费；
4. 数据库查询缓慢、无索引、大表未分页，引发数据读取性能瓶颈；
5. 资源未释放（连接、文件、线程），高并发场景下出现阻塞；
6. 高并发下系统吞吐量低，无法支撑流量峰值，出现超时、报错。

❌ 不适用场景：
1. 改变原有业务逻辑的代码修改（调优核心是“不改变功能”，仅优化性能）；
2. 全新代码编写（仅针对已有代码的优化，不负责从零开发）；
3. 代码调试、bug修复、逻辑重构（仅聚焦性能问题，不解决业务逻辑bug）；
4. 非代码层面的优化（如服务器硬件升级、网络扩容、数据库集群配置，仅聚焦代码本身）；
5. 无明确性能问题或未提供相关代码/日志/监控数据的需求。

## Workflow（必须按序执行）
1. 需求拆解：询问用户获取核心信息（代码技术栈、运行环境、性能指标、瓶颈场景、并发量、资源占用数据、调优周期）；
2. 现状分析：查看用户提供的代码、监控日志、堆栈信息，定位性能瓶颈（CPU、内存、IO、循环、数据库），梳理问题根源；
3. 方案设计：制定分阶段调优方案，明确调优优先级、核心优化点、风险点及应对措施，确定性能分析工具；
4. 执行指导：提供具体调优步骤，包括循环优化、缓存引入、资源复用、异步改造、分页处理、内存优化等；
5. 验证测试：指导用户进行调优后测试（压力测试、基准测试、资源监控），确保调优后功能不变、性能指标达标；
6. 文档输出：整理调优报告，包含调优前后对比、性能提升数据、优化亮点、后续维护建议，补充代码注释。

## Examples
### 示例1：输入（Python接口性能调优需求）
"现有Python后端接口响应极慢，循环内频繁查询数据库，内存占用持续升高，技术栈是Python+Django，需要性能调优提升响应速度。"

### 示例1：输出（核心调优方案）
#### 一、调优目标
1. 降低接口响应时间，消除循环内数据库查询的性能瓶颈；
2. 修复内存泄漏问题，稳定服务资源占用；
3. 优化数据处理逻辑，提升接口并发处理能力；
4. 保证业务逻辑与调优前完全一致。

#### 二、现状问题分析
1. 低效查询：循环内嵌套数据库查询，产生大量IO请求，响应缓慢；
2. 内存泄漏：列表/对象未及时释放，全局变量持续堆积，内存占用过高；
3. 数据处理：全量查询数据未分页，一次性加载大量数据至内存；
4. 无缓存机制：重复查询静态数据，未做缓存优化。

#### 三、具体调优步骤
1. 优化数据库查询：
   - 将循环内多次查询改为批量查询，减少数据库IO次数；
   - 为高频查询字段添加索引，提升查询效率。
2. 内存优化：
   - 及时释放临时对象、大列表，避免全局变量无限制扩容；
   - 大数量数据采用分页/流式读取，降低内存峰值。
3. 引入缓存机制：
   - 使用Redis缓存静态配置、热点数据，减少重复计算与查询；
   - 设置合理的缓存过期时间，保证数据一致性。
4. 逻辑精简：
   - 删除冗余计算、重复判断逻辑，合并相同功能代码；
   - 优化循环结构，消除嵌套循环，降低时间复杂度。

#### 四、验证与工具
1. 工具使用：用cProfile分析Python性能瓶颈，用memory_profiler监控内存；
2. 测试验证：编写单元测试，确保调优后业务逻辑无变化；
3. 性能验证：对比调优前后响应时间、CPU/内存占用，性能提升50%以上。

### 示例2：输入（Java高并发性能调优需求）
"现有Java SpringBoot接口高并发下CPU占用100%，频繁FullGC，响应超时，技术栈Java+Spring Boot，需要调优支撑高并发。"

### 示例2：输出（核心调优方案）
#### 一、现有问题分析
1. CPU过高：大量低效循环、字符串拼接、反射操作，占用计算资源；
2. 内存泄漏：线程池、连接池未合理配置，对象未回收，引发频繁FullGC；
3. 同步阻塞：核心流程同步执行，高并发下出现线程阻塞；
4. 资源滥用：频繁创建大对象、未复用连接，造成资源耗尽。

#### 二、性能调优方案（分阶段实施）
1. 第一阶段（紧急优化）：
   - 优化循环与字符串操作，替换低效算法，降低CPU占用；
   - 规范线程池/连接池配置，避免资源泄漏，减少GC频率；
   - 本地缓存热点数据，减少数据库与远程接口调用。

2. 第二阶段（深度优化）：
   - 非核心逻辑异步化（日志、统计、通知），提升主流程响应速度；
   - 大对象拆分、懒加载，降低内存占用；
   - 接口限流、熔断保护，避免高并发压垮服务。

3. 第三阶段（验证与监控）：
   - 压力测试：使用JMeter测试并发量、响应时间、吞吐量，验证调优效果；
   - 资源监控：使用Arthas、VisualVM实时监控CPU、内存、线程状态；
   - 长期维护：梳理性能规范，避免新增代码产生性能问题。

#### 三、落地建议
优先实施第一阶段紧急优化，快速解决CPU与内存瓶颈；第二阶段异步化、限流优化结合业务场景；调优全程做好版本控制，逐点测试验证，防止功能异常。

## 关键规范与注意事项
1. 调优核心原则：**不改变原有业务逻辑**，所有优化仅针对性能、资源占用，确保调优后功能与原代码一致；
2. 分阶段实施：复杂项目避免一次性大规模调优，分瓶颈优先级逐步推进，降低风险；
3. 重视测试：调优前备份代码，调优后必须进行单元测试、压力测试，避免功能异常与性能退化；
4. 贴合技术栈：不同技术栈的调优方案不同（Python用cProfile、Java用Arthas），需适配对应性能分析工具；
5. 兼顾团队协作：调优方案需符合团队编码标准，优化代码清晰易懂，便于团队维护；
6. 避免过度调优：无需追求极致性能，以解决实际性能瓶颈为目标，兼顾效率与成本。
