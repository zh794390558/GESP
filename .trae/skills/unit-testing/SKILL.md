# 单元测试Skill
## Overview
本技能专注于为各类项目（Web应用、后端服务、工具类函数、业务模块、接口方法等）提供全流程单元测试支持，
核心价值是在不修改原有业务逻辑的前提下，编写标准化测试用例、提升测试覆盖率、Mock外部依赖，保障代码稳定性，
适配不同技术栈（Java、Python、Go、JavaScript等）、不同测试框架（JUnit、pytest、testing、Jest）的单元测试需求，
兼顾测试规范性与覆盖率，为代码重构、功能迭代、问题排查提供可靠的测试保障。

## When to Use
✅ 适用场景：
1. 为函数、方法、业务模块从零编写单元测试用例，覆盖正常/异常/边界场景；
2. 提升项目单元测试覆盖率，修复失败测试用例，优化测试代码结构；
3. 单元测试依赖Mock（数据库、接口、第三方服务），解决测试环境依赖问题；
4. 代码重构/功能迭代后，编写回归测试用例，验证功能兼容性；
5. 测试代码不规范、结构混乱，需要统一整改与优化；
6. 单元测试运行失败、报错，需要排查调试测试用例。

❌ 不适用场景：
1. 全新业务代码开发（仅负责单元测试编写，不实现业务功能）；
2. 集成测试、接口测试、端到端测试、性能压测（仅聚焦单元级代码测试）；
3. 非代码类测试（产品功能测试、UI测试、硬件测试）；
4. 无源代码的单元测试需求（无法基于业务逻辑编写用例）；
5. 破坏原有业务逻辑的测试代码修改。

## Workflow（必须按序执行）
1. 需求拆解：询问用户获取核心信息（代码技术栈、测试框架、目标模块、覆盖率要求、依赖组件、项目结构）；
2. 现状分析：查看用户提供的源代码、模块逻辑、现有测试代码，识别未覆盖场景、依赖问题、测试缺陷；
3. 方案设计：制定测试用例设计方案，明确覆盖场景（正常/边界/异常）、Mock方案、覆盖率目标、测试结构；
4. 执行指导：提供可直接运行的测试代码，指导依赖Mock编写、用例分层、测试代码规范化；
5. 验证测试：指导运行单元测试、统计覆盖率、修复失败用例，确保测试用例有效且业务逻辑无误；
6. 文档输出：整理测试报告、覆盖率数据、用例说明、后续测试维护建议，规范测试代码管理。

## Examples
### 示例1：输入（Python单元测试编写需求）
"现有Python工具函数，用于计算订单价格，未编写单元测试，技术栈Python+pytest，需要覆盖所有场景并提升覆盖率。"

### 示例1：输出（核心测试方案）
#### 一、测试目标
1. 覆盖订单价格计算的正常场景、边界场景、异常场景；
2. 编写可直接运行的pytest测试用例；
3. 实现函数100%单元测试覆盖率，验证逻辑准确性。

#### 二、现状问题分析
1. 无单元测试用例，无法验证函数逻辑正确性；
2. 未考虑边界值（0元、折扣上限）、异常参数（负数、非数字）；
3. 无测试规范，后续迭代无法快速回归验证。

#### 三、具体测试步骤
1. 用例设计：
   - 正常场景：标准价格+折扣计算；
   - 边界场景：0元商品、最大折扣、最小折扣；
   - 异常场景：负数价格、非数字参数，校验异常抛出。
2. 编写测试代码：
   - 遵循pytest规范，函数命名以test_开头；
   - 分场景编写独立测试用例，提升可读性；
3. 依赖处理：无外部依赖，无需Mock，直接测试原生逻辑。
4. 执行验证：运行测试命令，查看通过率与覆盖率报告。

#### 四、验证与工具
1. 工具使用：使用pytest运行用例，pytest-cov统计覆盖率；
2. 测试验证：所有用例执行通过，逻辑与预期完全一致；
3. 覆盖率验证：实现目标函数100%行覆盖率、分支覆盖率。

### 示例2：输入（Java单元测试优化需求）
"现有SpringBoot服务层方法，依赖数据库查询，单元测试运行失败，技术栈Java+JUnit5+Mockito，需要Mock依赖并修复用例。"

### 示例2：输出（核心测试方案）
#### 一、现有问题分析
1. 单元测试直接依赖数据库，环境不兼容导致测试失败；
2. 未Mock外部依赖（Mapper接口），测试与环境强耦合；
3. 测试用例覆盖不全，缺少异常场景、空值场景测试。

#### 二、单元测试优化方案（分阶段实施）
1. 第一阶段（基础修复）：
   - 使用Mockito Mock数据库Mapper依赖，解除环境耦合；
   - 模拟数据库返回值，支持本地无环境运行测试；
   - 修复基础测试用例，确保正常场景执行通过。
2. 第二阶段（覆盖优化）：
   - 补充边界用例：空数据、参数异常、无权限场景；
   - 优化测试结构，分层编写用例，符合阿里巴巴开发手册；
   - 提升测试覆盖率至85%以上。
3. 第三阶段（验证与监控）：
   - 执行单元测试，生成Allure测试报告；
   - 集成覆盖率工具，持续监控测试完整性；
   - 绑定CI/CD，代码提交前自动运行测试用例。

#### 三、落地建议
优先Mock外部依赖，让单元测试可独立运行；分模块补充用例，逐步提升覆盖率；测试用例与业务代码同步维护，迭代后自动回归测试。

## 关键规范与注意事项
1. 测试核心原则：**不修改原有业务逻辑**，仅编写测试代码验证功能，保障业务代码稳定性；
2. 场景全覆盖：必须包含正常场景、边界场景、异常场景，杜绝仅测试基础逻辑的无效用例；
3. 解耦依赖：对数据库、第三方接口、外部服务必须使用Mock，单元测试不依赖运行环境；
4. 贴合技术栈：不同语言使用对应主流测试框架（Java=JUnit、Python=pytest、Go=testing）；
5. 可维护性：测试代码遵循与业务代码一致的规范，命名清晰、注释完整，便于团队维护；
6. 避免过度测试：聚焦核心业务逻辑与复杂函数，无需对简单get/set方法做无意义测试。
