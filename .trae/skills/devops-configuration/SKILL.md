# DevOps配置Skill
## Overview
本技能专注于为各类项目（Web应用、后端服务、移动端代码、工具类项目等）提供全流程DevOps配置支持，
核心价值是在不改变代码原有业务逻辑的前提下，搭建自动化流程、标准化部署环境、完善监控体系、提升交付效率，
适配不同技术栈（Java、Python、Go、JavaScript等）、不同部署环境（服务器、Docker、K8s、云平台）的DevOps配置需求，
兼顾配置效率与环境稳定性，避免配置过程中引发部署、运行故障。

## When to Use
✅ 适用场景：
1. 无自动化部署流程，需要搭建CI/CD流水线（GitLab CI、Jenkins、GitHub Actions）；
2. 项目未容器化，需要编写Dockerfile、docker-compose编排配置；
3. 部署环境混乱，需要标准化开发/测试/生产环境配置；
4. 无监控告警体系，需要搭建服务监控、日志收集、异常告警配置；
5. 部署操作重复繁琐，需要编写自动化脚本、一键部署方案；
6. 云原生部署需求，需要配置K8s资源、服务网格、弹性扩缩容。

❌ 不适用场景：
1. 改变原有业务逻辑的代码修改（配置核心是“不改变功能”，仅优化部署流程）；
2. 全新代码编写（仅针对已有项目的DevOps配置，不负责从零开发）；
3. 代码调试、bug修复、代码重构（仅聚焦环境与部署配置，不修改代码逻辑）；
4. 非DevOps类需求（产品设计、UI开发、业务逻辑实现）；
5. 无明确部署目标或未提供相关项目/环境信息的需求。

## Workflow（必须按序执行）
1. 需求拆解：询问用户获取核心信息（项目技术栈、部署环境、构建工具、发布流程、监控需求、配置周期）；
2. 现状分析：查看用户提供的项目结构、现有配置、部署方式，识别配置缺失、流程混乱、环境不规范问题；
3. 方案设计：制定分阶段配置方案，明确配置优先级、核心优化点、风险点及应对措施，确定DevOps工具链；
4. 执行指导：提供具体配置步骤，包括流水线编写、容器化配置、环境变量管理、监控部署、自动化脚本等；
5. 验证测试：指导用户进行配置后测试（构建测试、部署测试、运行测试），确保自动化流程、部署环境正常运行；
6. 文档输出：整理配置报告，包含配置前后对比、优化亮点、后续维护建议，补充部署文档、操作手册。

## Examples
### 示例1：输入（Python项目DevOps配置需求）
"现有Python Django项目，无自动化部署，手动上传服务器运行，需要搭建CI/CD+Docker容器化部署，技术栈Python+Django，实现自动化构建发布。"

### 示例1：输出（核心配置方案）
#### 一、配置目标
1. 搭建GitHub Actions CI/CD自动化流水线，实现代码提交自动构建；
2. 编写标准化Dockerfile，完成项目容器化打包；
3. 配置docker-compose编排服务，简化部署流程；
4. 统一环境变量管理，区分开发/生产环境。

#### 二、现状问题分析
1. 部署低效：手动打包、上传、启动服务，耗时易出错；
2. 无容器化：环境依赖不一致，易出现线上本地运行差异；
3. 无自动化：代码更新后需手动操作，无法快速迭代；
4. 配置混乱：环境变量硬编码，无统一管理规范。

#### 三、具体配置步骤
1. 容器化配置：
   - 编写Dockerfile，基于Python官方镜像，安装依赖、打包项目；
   - 编写docker-compose.yml，编排服务、端口映射、数据卷挂载。
2. CI/CD流水线配置：
   - 创建GitHub Actions工作流，配置代码拉取、镜像构建、镜像推送、服务器部署；
   - 设置触发规则，支持主分支自动发布、测试分支手动部署。
3. 环境标准化：
   - 分离.env环境配置文件，管理数据库、密钥等敏感信息；
   - 配置日志持久化、服务自启动、异常重启策略。
4. 自动化优化：
   - 编写一键启动/停止/重启脚本，简化运维操作；
   - 配置健康检查，保证服务稳定性。

#### 四、验证与工具
1. 工具使用：使用Docker做容器化，GitHub Actions做持续集成，docker-compose做编排；
2. 测试验证：代码提交自动触发流水线，镜像构建成功，服务正常启动运行；
3. 稳定性验证：容器自动重启，环境隔离，无依赖冲突问题。

### 示例2：输入（Java项目DevOps配置需求）
"现有Java SpringBoot项目，需要配置Jenkins流水线+K8s部署，无监控告警，技术栈Java+SpringBoot，实现云原生自动化运维。"

### 示例2：输出（核心配置方案）
#### 一、现有问题分析
1. 构建繁琐：手动Maven打包，部署流程复杂；
2. 无编排：单服务器部署，无法弹性扩缩容；
3. 无监控：服务状态、资源占用无监控，故障无法及时感知；
4. 无标准化：多环境部署配置不一致，维护成本高。

#### 二、DevOps配置方案（分阶段实施）
1. 第一阶段（基础配置）：
   - 配置Jenkins流水线，实现代码拉取、Maven构建、Docker镜像打包；
   - 编写K8s Deployment/Service配置文件，完成容器编排部署；
   - 配置ConfigMap/Secret，统一管理环境变量与敏感配置。

2. 第二阶段（运维优化）：
   - 搭建Prometheus+Grafana监控体系，监控CPU、内存、接口状态；
   - 配置ELK日志收集，统一管理服务日志，支持关键词检索；
   - 配置服务弹性扩缩容、健康检查、自愈策略。

3. 第三阶段（验证与监控）：
   - 流水线测试：代码提交自动构建部署，全流程无人工干预；
   - 监控验证：服务异常自动告警，资源指标可视化展示；
   - 文档梳理：形成DevOps配置手册，规范团队部署流程。

#### 三、落地建议
优先实施第一阶段基础配置，快速实现自动化部署；第二阶段监控配置逐步完善；配置过程做好版本控制，每次修改后验证环境，防止部署故障。

## 关键规范与注意事项
1. 配置核心原则：**不改变原有业务逻辑**，所有操作仅针对DevOps流程、环境、部署配置，确保项目功能不受影响；
2. 分阶段实施：复杂项目避免一次性大规模配置，分模块、分优先级逐步推进，降低风险；
3. 重视测试：配置前备份环境，配置后必须进行构建测试、部署测试、运行测试，避免环境故障；
4. 贴合技术栈：不同技术栈的构建工具不同（Java=Maven、Python=pip、Go=go build），需适配对应工具链；
5. 兼顾团队协作：配置方案需符合团队运维标准，便于团队成员理解和维护，统一部署规范；
6. 避免过度配置：无需追求极致复杂的云原生架构，以解决实际部署、效率问题为目标，兼顾简洁与实用。
