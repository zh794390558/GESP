# 文档生成Skill
## Overview
本技能专注于为各类项目（Web应用、后端服务、移动端代码、工具类项目等）提供全流程技术文档生成支持，
核心价值是在不改变代码原有业务逻辑的前提下，自动生成标准化文档、补全代码注释、规范文档结构，降低文档维护成本，
适配不同技术栈（Java、Python、Go、JavaScript等）、不同规模（小型工具、中型项目、大型系统）的文档生成需求，
兼顾文档规范性与可读性，让项目文档与代码同步更新，提升团队协作效率。

## When to Use
✅ 适用场景：
1. 项目无文档、注释缺失，需要快速生成README、接口文档、使用说明；
2. 代码注释不规范、格式混乱，需要统一补全、格式化函数/类注释；
3. API接口无文档，需要自动生成接口说明、参数、返回值、异常文档；
4. 项目部署、启动、配置无文档，需要生成部署手册、环境配置文档；
5. 技术文档版本混乱，需要统一规范格式、目录结构、内容模板；
6. 新人上手困难，需要生成项目架构、业务流程、代码说明文档。

❌ 不适用场景：
1. 改变原有业务逻辑的代码修改（生成文档核心是“不改变功能”，仅补充文档）；
2. 全新代码编写（仅针对已有代码生成文档，不负责从零开发）；
3. 代码调试、bug修复、代码重构（仅聚焦文档生成，不修改代码逻辑）；
4. 非技术类文档（如产品文档、市场文案、行政文档）；
5. 无明确代码/项目、无业务说明的模糊文档生成需求。

## Workflow（必须按序执行）
1. 需求拆解：询问用户获取核心信息（代码技术栈、文档类型、项目结构、业务场景、文档格式要求、输出路径）；
2. 现状分析：查看用户提供的代码片段/项目结构，识别文档缺失、注释不规范、格式混乱问题，梳理文档缺口；
3. 方案设计：制定分阶段文档生成方案，明确文档优先级、核心内容、格式规范、模板样式，确定文档生成工具；
4. 执行指导：提供具体文档生成步骤，包括注释补全、文档编写、格式规范、目录生成、内容填充等；
5. 验证测试：指导用户进行文档校验，确保文档内容与代码/业务一致，格式规范、可读性达标；
6. 文档输出：整理最终文档包，包含文档目录、注释规范、更新说明、后续维护建议，同步更新代码注释。

## Examples
### 示例1：输入（项目文档生成需求）
"现有Python后端项目，无README文档、代码注释缺失，技术栈是Python+Django，需要生成完整项目文档和代码注释。"

### 示例1：输出（核心文档生成方案）
#### 一、生成目标
1. 生成标准化项目README文档，包含项目介绍、启动方式、环境配置、目录结构；
2. 补全所有核心函数、类的代码注释，符合Python docstring规范；
3. 统一文档格式，提升项目可读性和新人上手效率。

#### 二、现状问题分析
1. 文档缺失：无项目README，无使用说明、部署说明；
2. 注释缺失：核心函数、类、接口无注释，参数/返回值未说明；
3. 格式混乱：无统一文档规范，后续维护困难；
4. 信息缺失：无项目目录结构、业务流程说明。

#### 三、具体生成步骤
1. 生成README文档：
   - 在项目根目录生成README.md，包含项目概述、环境依赖、启动命令、目录结构；
   - 补充常见问题、更新日志、贡献规范，完善项目说明。
2. 补全代码注释：
   - 函数/类添加规范docstring，说明功能、参数、返回值、异常；
   - 复杂业务逻辑添加行注释，解释核心流程；
   - 接口方法标注请求方式、参数、返回格式。
3. 规范文档结构：
   - 统一文档标题、列表、代码块格式；
   - 生成文档目录，支持快速跳转查阅。
4. 同步更新维护：
   - 文档与代码绑定，迭代时同步更新；
   - 标注文档版本、最后更新时间。

#### 四、验证与工具
1. 工具使用：用pdoc自动生成Python API文档，用markdownlint校验文档格式；
2. 内容验证：文档与代码逻辑完全一致，注释清晰无歧义；
3. 可读性验证：新人可通过文档独立完成项目部署、代码理解。

### 示例2：输入（接口文档生成需求）
"现有Java SpringBoot接口项目，无API文档，接口参数、返回值不清晰，技术栈Java+SpringBoot，需要生成接口文档。"

### 示例2：输出（核心文档生成方案）
#### 一、现有问题分析
1. 接口无文档：无请求方式、参数、返回值、异常说明；
2. 对接成本高：前端、第三方对接需要反复沟通接口定义；
3. 无统一规范：接口注释格式混乱，无标准化文档；
4. 无版本管理：接口迭代后文档无法同步更新。

#### 二、文档生成方案（分阶段实施）
1. 第一阶段（基础生成）：
   - 基于SpringBoot代码生成Swagger/OpenAPI接口文档；
   - 补全接口注解，标注接口名称、请求参数、响应格式、错误码；
   - 生成可视化接口页面，支持在线调试。
2. 第二阶段（规范优化）：
   - 统一接口文档格式，按模块拆分接口分类；
   - 补充接口业务说明、调用流程、注意事项；
   - 生成离线Markdown/PDF文档，适配离线使用。
3. 第三阶段（验证与监控）：
   - 校验文档与接口代码一致性，避免文档过期；
   - 集成自动构建，代码更新后自动同步接口文档；
   - 形成团队文档规范，后续接口统一遵循。

#### 三、落地建议
优先生成可视化接口文档，快速满足对接需求；分模块补全注释，避免一次性大规模修改；文档与代码同步提交，确保文档实时生效。

## 关键规范与注意事项
1. 文档生成核心原则：**不改变原有业务逻辑**，所有操作仅针对文档补充、格式规范，确保代码功能不受影响；
2. 分阶段实施：复杂项目避免一次性全量生成，分模块、分优先级逐步推进，降低风险；
3. 重视校验：文档生成后必须核对代码、业务逻辑，避免文档与代码不一致，确保准确性；
4. 贴合技术栈：不同技术栈的文档规范不同（Python遵循Google注释、Java遵循JavaDoc），需适配对应规范；
5. 兼顾团队协作：文档方案需符合团队编码标准，统一格式、目录、注释风格，便于全员维护；
6. 避免过度生成：无需生成无意义注释、冗余文档，以解决实际协作、维护问题为目标，兼顾简洁与实用。
