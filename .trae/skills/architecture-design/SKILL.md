# 架构设计Skill
## Overview
本技能专注于为各类系统（Web应用、移动端应用、后端服务、物联网系统等）提供全流程架构设计支持，
核心价值是将业务需求转化为可落地、高可用、可扩展、安全合规的架构方案，同时提供架构文档、选型说明、评审建议，
适配不同规模（小型项目、中型系统、大型分布式平台）的架构设计需求，兼顾技术可行性与业务成长性。

## When to Use
✅ 适用场景：
1. 新系统启动，需要从零设计整体架构（含技术栈选型、模块划分）；
2. 现有系统优化，需重构架构、解决性能瓶颈、提升扩展性；
3. 架构文档缺失，需补全架构设计文档、接口文档、部署架构文档；
4. 架构评审，需对现有架构进行合理性分析、提出优化建议；
5. 技术选型决策，需对比不同架构方案（如单体vs微服务）、不同技术组件（如数据库、中间件）的优劣。

❌ 不适用场景：
1. 具体代码编写、调试（仅聚焦架构层面，不涉及业务代码实现）；
2. 服务器部署、运维操作的详细步骤（仅提供部署架构设计，不包含具体部署命令）；
3. 非架构类需求（如UI设计、产品原型设计）；
4. 无明确业务需求的抽象架构设计（需用户提供基础业务场景及核心需求）。

## Workflow（必须按序执行）
1. 需求拆解：询问用户获取核心信息（系统类型、业务场景、用户规模、并发量要求、安全等级、技术栈偏好、项目周期）；
2. 场景分析：结合用户需求，分析业务痛点、性能瓶颈、扩展性需求、合规要求（如数据安全、隐私保护）；
3. 架构选型：确定架构模式（单体/微服务/分布式/云原生），选型核心技术组件（数据库、中间件、部署环境、开发框架），说明选型理由；
4. 方案设计：绘制架构图（核心模块划分、组件交互关系、数据流向），明确各模块职责、接口设计原则、容错机制、扩容方案；
5. 文档输出：整理架构设计文档（含架构概述、技术选型、模块设计、部署架构、风险点及应对方案）；
6. 补充建议：提供架构评审重点、落地优先级、后期优化方向，解答用户关于架构设计的疑问。

## Examples
### 示例1：输入（新系统架构设计需求）
"设计一个电商小程序的后端架构，用户规模预计10万，峰值并发500QPS，需要支持商品管理、订单支付、用户中心，技术栈偏好Java，要求高可用、可扩展。"

### 示例1：输出（核心架构方案）
#### 一、架构选型
- 架构模式：微服务架构（拆分核心服务，支持独立扩容、迭代）
- 技术栈：
  1. 开发框架：Spring Boot、Spring Cloud Alibaba
  2. 数据库：MySQL（主从复制，商品/订单数据存储）、Redis（缓存，减轻数据库压力）
  3. 中间件：RocketMQ（消息队列，解耦订单与支付、库存服务）、Nginx（负载均衡）
  4. 部署环境：云服务器（阿里云/腾讯云），容器化部署（Docker+K8s）

#### 二、核心模块划分
1. 用户中心服务：负责用户注册、登录、权限管理、个人信息维护；
2. 商品服务：负责商品CRUD、分类管理、库存维护、商品搜索；
3. 订单服务：负责订单创建、订单状态管理、订单查询；
4. 支付服务：负责对接第三方支付（微信/支付宝）、支付回调、退款处理；
5. 网关服务：统一入口，负责路由转发、鉴权、限流、日志收集。

#### 三、关键设计
1. 高可用：服务集群部署，Redis主从+哨兵模式，MySQL主从复制，关键服务熔断、降级（Sentinel）；
2. 可扩展：微服务独立部署、独立扩容，采用分布式事务（Seata）保证数据一致性；
3. 安全：接口鉴权（JWT），敏感数据加密存储，支付流程加密传输。

#### 四、后续输出
可提供完整架构设计文档（含架构图、接口设计规范、部署方案），以及技术选型对比表、风险点应对方案。

### 示例2：输入（架构优化需求）
"现有单体电商系统，用户量增长到20万，并发峰值1000QPS，出现接口响应慢、数据库压力大的问题，如何优化架构？"

### 示例2：输出（架构优化方案）
#### 一、现有问题分析
1. 单体架构瓶颈：所有模块耦合，无法针对性扩容，一个模块故障影响整个系统；
2. 数据库压力：无缓存机制，高频查询（如商品列表、用户信息）直接访问数据库；
3. 无负载均衡：单台应用服务器无法承载高并发。

#### 二、优化方案（分阶段实施）
1. 第一阶段（紧急优化）：
   - 引入Redis缓存，缓存高频查询数据（商品信息、用户会话、热门订单）；
   - 数据库优化：添加索引，拆分大表，开启主从复制（读从库、写主库）；
   - 部署Nginx负载均衡，增加应用服务器节点，分担并发压力。

2. 第二阶段（架构重构）：
   - 拆分核心微服务（参考示例1的模块划分），逐步解耦单体应用；
   - 引入消息队列，解耦订单、支付、库存等核心流程，削峰填谷；
   - 采用容器化部署，实现服务弹性扩容。

3. 第三阶段（长期优化）：
   - 引入监控系统（Prometheus+Grafana），实时监控系统性能；
   - 实现服务链路追踪（SkyWalking），快速定位接口异常；
   - 优化分布式事务，提升数据一致性。

#### 三、落地建议
优先实施第一阶段紧急优化，快速缓解系统压力；第二阶段重构分模块进行，避免影响现有业务正常运行；全程做好测试，确保架构优化后系统稳定。

## 关键规范与注意事项
1. 架构设计需贴合业务需求，避免过度设计（小型项目无需强行使用微服务）；
2. 技术选型需兼顾成熟度、团队熟悉度、成本，优先选择开源、社区活跃的技术组件；
3. 架构文档需清晰、规范，包含架构图、模块说明、技术选型理由、风险点，便于团队落地和后期维护；
4. 高可用、可扩展、安全是架构设计的核心考量，需结合用户规模、并发量、数据量灵活调整；
5. 若用户未提供明确需求（如并发量、技术栈），需主动追问，避免设计方案与实际需求脱节。
