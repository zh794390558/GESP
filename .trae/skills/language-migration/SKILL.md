# 语言迁移Skill
## Overview
本技能专注于为各类项目（Web应用、后端服务、移动端代码、工具类项目等）提供全流程代码语言迁移支持，
核心价值是在**完全保留原有业务逻辑**的前提下，完成代码跨语言转换、语法适配、依赖对齐、功能验证，
适配主流语言迁移场景（Python↔Go、Java↔Kotlin、JS↔TS、PHP↔Python等）、不同规模（小型工具、中型项目、大型系统）的迁移需求，
兼顾迁移效率与代码稳定性，确保迁移后代码功能与原逻辑完全一致，无业务偏差。

## When to Use
✅ 适用场景：
1. 技术栈统一，需要将旧语言代码迁移为团队标准语言（如PHP迁移Python、Java迁移Go）；
2. 旧语言生态废弃、维护成本高，需迁移至现代编程语言（如Python2迁移Python3、JS迁移TypeScript）；
3. 性能/架构升级，需要将脚本语言迁移为编译型语言（Python迁移Go、Ruby迁移Java）；
4. 语法版本升级，低版本语法迁移至高版本规范（Java8迁移Java17、ES5迁移ES6）；
5. 跨平台代码移植，需要将单一语言代码转换为多语言兼容版本；
6. 依赖库不兼容，需通过语言迁移适配新框架、新中间件、新服务。

❌ 不适用场景：
1. 改变原有业务逻辑的代码修改（迁移核心是“不改变功能”，仅转换语言语法）；
2. 全新代码编写（仅针对已有代码的迁移转换，不负责从零开发）；
3. 代码调试、bug修复、性能优化（仅聚焦语言迁移，不解决逻辑层面问题）；
4. 非代码层面的迁移（服务器迁移、数据库迁移、配置迁移，仅聚焦代码本身）；
5. 无明确源语言/目标语言或未提供相关代码片段/代码现状的需求。

## Workflow（必须按序执行）
1. 需求拆解：询问用户获取核心信息（源语言、目标语言、代码规模、业务场景、框架依赖、迁移周期、验证标准）；
2. 现状分析：查看用户提供的代码片段/项目结构，识别语法差异、依赖冲突、API不兼容、逻辑难点，梳理迁移风险；
3. 方案设计：制定分阶段迁移方案，明确迁移优先级、核心转换点、风险点及应对措施，确定迁移工具（语法转换、校验工具）；
4. 执行指导：提供具体迁移步骤，包括语法转换、依赖替换、API对齐、类型适配、代码规范化等；
5. 验证测试：指导用户进行迁移后测试（单元测试、集成测试、回归测试），确保迁移后代码功能与原有一致，运行稳定性达标；
6. 文档输出：整理迁移报告，包含迁移前后对比、语法差异、适配亮点、后续维护建议，补充迁移后代码注释、文档说明。

## Examples
### 示例1：输入（工具函数语言迁移需求）
"现有Python工具函数，需要迁移为Go语言，保持原有计算逻辑不变，技术栈Python3.10转Go1.21，需要完成语法转换并保证功能一致。"

### 示例1：输出（核心迁移方案）
#### 一、迁移目标
1. 完成Python到Go的语法精准转换，不修改原有计算逻辑；
2. 对齐Go语言编码规范，符合Go官方标准；
3. 补充必要注释，提升迁移后代码可读性；
4. 验证功能一致性，确保输入输出与原Python代码完全一致。

#### 二、现状问题分析
1. 语法差异：Python动态类型与Go静态类型不兼容，需补充类型声明；
2. 函数差异：Python内置函数与Go标准库API不同，需替换等效实现；
3. 命名不规范：Python蛇形命名需转换为Go驼峰命名规范；
4. 依赖差异：Python第三方库需替换为Go等效依赖包。

#### 三、具体迁移步骤
1. 语法基础转换：
   - 静态类型声明：为变量、函数参数、返回值添加Go类型定义；
   - 命名规范转换：蛇形命名转为驼峰命名，符合Go编码规范；
   - 流程控制转换：if/for/异常处理转换为Go原生语法。
2. 依赖与API替换：
   - 替换Python内置函数为Go标准库等效实现；
   - 无原生支持的逻辑，封装等效工具函数。
3. 补充注释：
   - 函数头部添加Go文档注释，说明功能、参数、返回值；
   - 复杂逻辑处添加行注释，对齐原代码逻辑说明；
   - 标注迁移记录、语言适配说明。
4. 逻辑对齐：
   - 严格保留原有业务逻辑，不新增、不删减功能；
   - 边界值、异常处理逻辑与原代码完全一致。

#### 四、验证与工具
1. 工具使用：用gofmt格式化代码，用golint检测代码规范；
2. 测试验证：编写单元测试，覆盖所有输入场景，确保输出结果一致；
3. 稳定性验证：运行迁移后代码，无崩溃、无语法错误，性能符合预期。

### 示例2：输入（后端接口语言迁移需求）
"现有Java SpringBoot接口代码，需要迁移为Kotlin，存在大量Java语法和依赖，技术栈Java8+SpringBoot转Kotlin+SpringBoot，需要迁移并保证接口功能不变。"

### 示例2：输出（核心迁移方案）
#### 一、现有问题分析
1. 语法差异：Java空值处理、数据类、Lambda表达式与Kotlin不兼容；
2. 冗余代码：Java样板代码需简化为Kotlin简洁语法；
3. 依赖兼容：Java依赖库需适配Kotlin调用方式；
4. 空安全：Java无空安全机制，迁移后需适配Kotlin空安全规范。

#### 二、语言迁移方案（分阶段实施）
1. 第一阶段（基础转换）：
   - Java基础语法转换为Kotlin语法，保留原有业务逻辑；
   - 数据类替换为Kotlin data class，消除冗余Getter/Setter；
   - 空值处理适配Kotlin空安全机制，避免空指针异常。
2. 第二阶段（优化适配）：
   - 使用Kotlin扩展函数、拓展属性简化代码结构；
   - 依赖注入、接口定义适配Kotlin SpringBoot规范；
   - 统一代码风格，消除Java与Kotlin混合编码混乱问题。
3. 第三阶段（验证与监控）：
   - 接口回归测试，确保请求参数、返回值、业务逻辑与原Java一致；
   - 运行测试，验证Kotlin代码无兼容性问题、无性能退化；
   - 代码复盘，梳理迁移规范，形成团队语言迁移标准。

#### 三、落地建议
优先实施第一阶段基础转换，快速完成语法对齐；第二阶段优化结合业务场景，避免过度简化；迁移过程中做好版本控制，每次迁移后及时测试，防止引入功能偏差。

## 关键规范与注意事项
1. 迁移核心原则：**完全不改变原有业务逻辑**，所有操作仅针对语言语法、依赖适配，确保迁移后功能与原代码一致；
2. 分阶段实施：复杂项目避免一次性大规模迁移，分模块、分优先级逐步推进，降低风险；
3. 重视测试：迁移前备份代码，迁移后必须进行单元测试、集成测试、回归测试，避免功能异常；
4. 贴合语言规范：不同语言的编码规范不同（如Go驼峰、Python蛇形、Kotlin空安全），需适配对应语言标准；
5. 兼顾团队协作：迁移方案需符合团队技术标准，便于团队成员理解和维护，避免个人风格主导；
6. 避免过度优化：无需追求极致简洁代码，迁移以保证功能一致、稳定运行为核心目标，兼顾效率与成本。
