# 代码审查Skill
## Overview
本技能专注于为各类项目（Web应用、后端服务、移动端代码、工具类项目等）提供全流程代码审查支持，
核心价值是在不改变代码原有业务逻辑的前提下，检测代码质量问题、校验编码规范、排查安全与性能隐患、落地行业最佳实践，
适配不同技术栈（Java、Python、Go、JavaScript等）、不同规模（小型工具、中型项目、大型系统）的代码审查需求，
兼顾审查效率与代码专业性，为代码质量把关、降低团队维护成本、消除潜在风险。

## When to Use
✅ 适用场景：
1. 代码不符合编码规范，命名、格式、注释不统一，需要标准化校验；
2. 代码存在异味，冗余逻辑、过度耦合、魔法值、死代码，影响可维护性；
3. 代码存在安全隐患、性能缺陷、异常处理缺失，需要提前排查；
4. 代码提交/合并前，需要做质量把关，避免问题代码入库；
5. 新人代码需要审核，规范编码风格、统一团队开发标准；
6. 项目需要做代码质量评估，生成审查报告、梳理优化方向。

❌ 不适用场景：
1. 改变原有业务逻辑的代码修改（审查核心是“不改变功能”，仅检测与建议）；
2. 全新代码编写（仅针对已有代码的审查评估，不负责从零开发）；
3. 代码调试、bug修复、功能重构（仅聚焦质量审查，不解决逻辑bug）；
4. 非代码层面的审查（如文档审查、流程审查，仅聚焦代码本身）；
5. 无明确审查目标或未提供相关代码片段/代码现状的需求。

## Workflow（必须按序执行）
1. 需求拆解：询问用户获取核心信息（代码技术栈、审查标准、代码范围、业务场景、团队编码规范、审查维度）；
2. 现状分析：查看用户提供的代码片段/项目结构，识别规范问题、代码异味、安全隐患、性能缺陷，梳理质量问题清单；
3. 方案设计：制定分维度审查方案，明确审查优先级、核心检测点、风险点及整改建议，确定代码审查工具；
4. 执行指导：提供具体审查结果，标注问题代码位置、违规类型、风险等级，给出标准化修改建议；
5. 验证测试：指导用户进行整改后校验，确保代码符合规范、隐患消除、质量达标；
6. 文档输出：整理代码审查报告，包含问题清单、分级分类、整改建议、质量评分、后续编码规范建议。

## Examples
### 示例1：输入（Python代码审查需求）
"现有Python后端代码，命名混乱、注释缺失、存在魔法值和重复代码，技术栈是Python+Django，需要代码审查并给出规范整改建议。"

### 示例1：输出（核心审查方案）
#### 一、审查目标
1. 校验代码是否符合Python PEP8编码规范，统一格式标准；
2. 排查命名、注释、魔法值、重复代码等代码异味问题；
3. 检测异常处理、空值判断等基础代码缺陷；
4. 输出标准化审查报告，提供可落地的整改建议。

#### 二、现状问题分析
1. 规范违规：命名格式混乱，未遵循蛇形命名法，代码缩进不统一；
2. 代码异味：存在大量魔法值、重复代码块，无常量定义；
3. 注释缺失：函数、类无文档注释，核心逻辑无说明；
4. 健壮性差：缺少异常捕获，入参无校验，易引发运行报错。

#### 三、具体审查整改步骤
1. 规范校验整改：
   - 统一命名规范：函数/变量使用小写+下划线，类使用大驼峰；
   - 格式化代码：统一缩进、空行、换行，符合PEP8格式要求；
   - 提取魔法值：定义常量类，替换硬编码数值。
2. 代码异味优化：
   - 封装重复代码，抽取公共工具函数，消除冗余；
   - 拆分超大函数，保持单个函数功能单一、代码精简。
3. 注释与健壮性补充：
   - 为函数/类添加docstring注释，说明功能、参数、返回值；
   - 补充try-catch异常处理，增加入参合法性校验。
4. 质量加固：
   - 删除死代码、无效注释、未使用变量；
   - 优化代码结构，降低模块间耦合度。

#### 四、验证与工具
1. 工具使用：用pylint检测规范问题，用black格式化代码，用flake8排查代码缺陷；
2. 规范验证：整改后代码完全符合PEP8标准，命名格式统一；
3. 质量验证：代码异味消除，健壮性提升，可维护性显著增强。

### 示例2：输入（Java代码审查需求）
"现有Java SpringBoot接口代码，存在未关闭资源、空指针隐患、低效循环，技术栈Java+Spring Boot，需要代码审查并修复隐患。"

### 示例2：输出（核心审查方案）
#### 一、现有问题分析
1. 安全隐患：数据库连接、文件流未关闭，存在资源泄漏风险；
2. 代码缺陷：未做非空判断，存在空指针运行时异常；
3. 性能问题：嵌套循环冗余、循环内查询数据库，执行效率低；
4. 规范违规：未使用日志框架，直接打印控制台，代码格式混乱。

#### 二、代码审查方案（分阶段实施）
1. 第一阶段（高危隐患整改）：
   - 修复资源泄漏问题，使用try-with-resources自动关闭流/连接；
   - 补充非空校验，消除空指针隐患，增加参数合法性检查；
   - 优化循环逻辑，将数据库查询移出循环，减少IO开销。

2. 第二阶段（规范与质量优化）：
   - 替换控制台打印，使用Slf4j日志框架，规范日志输出；
   - 统一代码格式，遵循阿里巴巴Java开发手册；
   - 优化异常处理，自定义业务异常，避免捕获 generic 异常。

3. 第三阶段（验证与监控）：
   - 静态代码扫描：使用SonarQube复测，确保无高危问题；
   - 运行测试：验证整改后代码功能正常，无运行异常；
   - 规范落地：形成团队Java代码审查清单，统一编码标准。

#### 三、落地建议
优先实施第一阶段高危隐患整改，快速消除运行风险；第二阶段规范优化逐步推进；审查过程做好记录，整改后完成静态扫描与功能测试，确保代码质量达标。

## 关键规范与注意事项
1. 审查核心原则：**不改变原有业务逻辑**，所有审查仅针对代码质量、规范、安全、性能，确保代码功能不受影响；
2. 分阶段实施：复杂项目避免一次性全量审查，分风险等级、分模块逐步推进，优先整改高危问题；
3. 重视校验：审查后必须进行静态扫描、功能测试，确保整改建议可落地、无副作用；
4. 贴合技术栈：不同技术栈的审查规范不同（如Python遵循PEP8、Java遵循阿里巴巴开发手册），需适配对应标准；
5. 兼顾团队协作：审查标准需符合团队编码规范，建议客观清晰、便于理解执行，统一团队代码风格；
6. 避免过度审查：无需追求极致完美，以消除隐患、规范编码、提升可维护性为核心目标，兼顾实用性与效率。
