# 安全审计Skill
## Overview
本技能专注于为各类项目（Web应用、后端服务、移动端代码、工具类项目等）提供全流程代码安全审计支持，
核心价值是在不改变代码原有业务逻辑的前提下，检测安全漏洞、排查风险隐患、加固代码防护、符合安全合规要求，
适配不同技术栈（Java、Python、Go、JavaScript等）、不同规模（小型工具、中型项目、大型系统）的安全审计需求，
兼顾审计效率与代码稳定性，在修复漏洞的同时避免引入新的bug。

## When to Use
✅ 适用场景：
1. 代码存在安全漏洞，包括SQL注入、XSS跨站、CSRF、文件上传、命令执行等高危漏洞；
2. 敏感信息泄露，硬编码密钥、密码、token、身份证、手机号等隐私数据未加密；
3. 权限控制缺失，越权访问、未鉴权接口、用户数据未隔离等权限漏洞；
4. 第三方依赖存在高危漏洞，依赖包版本过时、存在已知安全风险；
5. 代码安全规范缺失，日志泄露敏感信息、输入未校验、加密算法不安全；
6. 项目需合规整改，满足等保、隐私合规、企业安全编码标准要求。

❌ 不适用场景：
1. 改变原有业务逻辑的代码修改（审计核心是“不改变功能”，仅修复安全漏洞）；
2. 全新代码编写（仅针对已有代码的安全检测，不负责从零开发）；
3. 代码重构、性能优化、功能调试（仅聚焦代码安全，不优化结构与性能）；
4. 非代码层面的安全（服务器防护、网络防火墙、数据库权限，仅聚焦代码本身）；
5. 无明确审计目标或未提供相关代码片段/项目结构的需求。

## Workflow（必须按序执行）
1. 需求拆解：询问用户获取核心信息（代码技术栈、审计目标、代码规模、业务场景、安全合规标准、审计范围）；
2. 现状分析：查看用户提供的代码片段/项目结构，识别安全漏洞（高危/中危/低危），梳理安全风险清单；
3. 方案设计：制定分阶段审计修复方案，明确漏洞优先级、核心加固点、风险点及应对措施，确定安全审计工具；
4. 执行指导：提供具体修复步骤，包括漏洞修复、敏感信息加密、权限加固、依赖升级、输入校验等；
5. 验证测试：指导用户进行安全复测、渗透测试，确保漏洞完全修复，安全防护达标；
6. 文档输出：整理安全审计报告，包含漏洞清单、修复前后对比、加固亮点、后续安全维护建议。

## Examples
### 示例1：输入（代码安全审计需求）
"现有Python后端代码，存在SQL注入风险，硬编码数据库密码，技术栈是Python+Django，需要安全审计并修复漏洞。"

### 示例1：输出（核心审计方案）
#### 一、审计目标
1. 修复SQL注入高危漏洞，保障数据库安全；
2. 清除硬编码敏感信息，符合数据安全规范；
3. 补充输入校验与安全防护，提升代码安全性；
4. 形成安全编码规范，避免后续新增漏洞。

#### 二、现状问题分析
1. SQL注入漏洞：直接拼接SQL语句，未使用参数化查询；
2. 敏感信息泄露：数据库账号、密码硬编码在代码中，无加密存储；
3. 输入校验缺失：用户请求参数未过滤，存在安全风险；
4. 安全规范缺失：无日志脱敏、无异常安全处理。

#### 三、具体修复步骤
1. 修复SQL注入漏洞：
   - 替换原生SQL拼接，使用Django ORM/参数化查询；
   - 禁止直接拼接用户输入参数，杜绝注入风险。
2. 清除硬编码信息：
   - 敏感配置移入环境变量/配置中心，加密存储；
   - 使用密钥管理工具，禁止明文密码提交至代码库。
3. 补充安全防护：
   - 增加用户输入参数校验、过滤特殊字符；
   - 日志脱敏处理，不打印敏感信息。
4. 权限加固：
   - 最小化数据库账号权限，禁止超权访问；
   - 接口添加身份鉴权，防止未授权访问。

#### 四、验证与工具
1. 工具使用：用bandit扫描Python安全漏洞，用safety检测依赖风险；
2. 安全验证：渗透测试无注入漏洞，敏感信息无明文泄露；
3. 合规验证：符合企业安全编码规范与数据隐私要求。

### 示例2：输入（接口安全审计需求）
"现有Java接口代码，存在越权访问、XSS漏洞，依赖包存在高危漏洞，技术栈Java+Spring Boot，需要审计加固。"

### 示例2：输出（核心审计方案）
#### 一、现有问题分析
1. 越权访问：接口未校验用户权限，可随意查看他人数据；
2. XSS跨站漏洞：前端输出未转义，用户输入直接渲染页面；
3. 依赖漏洞：Fastjson、Log4j等组件版本过时，存在远程执行漏洞；
4. 防护缺失：未开启CORS、CSP安全策略，接口无限流防护。

#### 二、安全审计方案（分阶段实施）
1. 第一阶段（紧急修复）：
   - 修复越权漏洞，添加用户数据权限校验、接口鉴权；
   - 升级高危依赖包版本，修复已知远程执行漏洞；
   - 增加XSS防护，参数转义、开启浏览器安全策略。
2. 第二阶段（深度加固）：
   - 开启接口限流、防刷防护，避免暴力攻击；
   - 配置CORS、CSP安全头，强化浏览器防护；
   - 敏感数据传输加密，接口返回值脱敏处理。
3. 第三阶段（验证与监控）：
   - 渗透测试复测，确保所有漏洞完全修复；
   - 接入安全扫描工具，持续监控代码安全风险；
   - 形成团队安全编码规范，全员遵循执行。

#### 三、落地建议
优先实施第一阶段紧急修复，快速封堵高危漏洞；第二阶段加固结合业务场景，避免过度防护；审计过程做好版本控制，修复后及时复测，防止漏洞复现。

## 关键规范与注意事项
1. 审计核心原则：**不改变原有业务逻辑**，所有修复仅针对安全漏洞与防护加固，确保审计后功能与原代码一致；
2. 分阶段实施：复杂项目避免一次性大规模修复，分漏洞等级、分模块逐步推进，降低风险；
3. 重视复测：审计前备份代码，修复后必须进行安全复测、渗透测试，避免漏洞未修复或引入新风险；
4. 贴合技术栈：不同技术栈的安全规范不同（Python用bandit、Java用sonarqube），需适配对应安全工具；
5. 兼顾团队协作：审计方案需符合企业安全标准，便于团队成员理解和维护，统一安全编码规范；
6. 避免过度加固：无需追求极致安全防护，以解决实际安全风险为目标，兼顾安全性与业务可用性。
